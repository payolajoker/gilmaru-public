# 길마루 오픈 라이선스 전략

이 문서는 길마루를 공익 오픈소스, 카피레프트 성향 프로젝트로 전환하기 위한 권장 라이선스 구조를 정리한다.

중요:

- 이 문서는 `법률 자문이 아니라 운영 초안`이다.
- 실제 라이선스 변경 전에는 모든 데이터와 자산의 출처를 다시 점검해야 한다.
- 현재 저장소의 `package.json` 라이선스는 `ISC`이며, 이 문서는 향후 전환 전략을 제안하는 문서다.

## 1. 왜 단일 라이선스로 끝내면 안 되는가

길마루는 단순 코드 저장소가 아니다.
한 저장소 안에 서로 성격이 다른 자산이 섞여 있다.

- 앱 소스코드
- 단어 데이터
- 외부 어휘 파일
- 문서
- 발표 자료
- 이미지와 스크린샷

이 모든 것을 한 장짜리 LICENSE 파일 하나로 처리하면 오히려 권리관계가 불명확해진다.
따라서 `자산 종류별 분리 라이선스`가 맞다.

## 2. 권장 최종 구조

### 코드: `AGPL-3.0-or-later`

대상:

- `app.js`
- `gilmaru_core.js`
- `scripts/`
- 테스트 코드
- 빌드 설정

이유:

- 길마루는 웹앱 형태로 배포될 가능성이 높다.
- 일반 GPL보다 AGPL이 네트워크 배포 상황까지 더 강하게 환원 의무를 만든다.
- 카피레프트 성향을 분명하게 가져가기에 적합하다.

### 커뮤니티 포인트 데이터: `ODbL-1.0`

대상:

- 길마루 포인트 데이터셋
- 지역별 공간 정보 모음
- 공개 배포용 위치 포인트 팩

이유:

- 위치 데이터와 공간 데이터에는 ODbL이 잘 맞는다.
- 재배포와 파생 데이터의 공유 조건을 명확히 두기 좋다.

### 문서와 가이드: `CC BY-SA 4.0`

대상:

- `docs/`
- 안내 문안
- 운영 가이드
- 교육 자료

이유:

- 문서류에는 코드 라이선스보다 CC 계열이 더 이해하기 쉽다.
- `BY-SA`는 카피레프트 느낌을 유지하면서 재사용도 허용한다.

### 로고와 프로젝트 이름: 별도 상표 정책

대상:

- `Gilmaru`, `길마루` 명칭
- 로고
- 공식 시각 정체성

이유:

- 오픈소스라고 해서 이름과 로고까지 완전히 무제한으로 열 필요는 없다.
- 악의적 사칭과 혼동을 막기 위한 별도 상표 정책이 필요하다.

### 예시 데이터와 샘플 텍스트: `CC BY-SA 4.0` 또는 `CC0-1.0`

대상:

- 데모용 포인트
- 샘플 설명문
- 템플릿 예시

권장:

- 공동 저작을 강하게 유지하려면 `CC BY-SA 4.0`
- 최대한 퍼뜨리는 데 집중하려면 `CC0-1.0`

## 3. 당장 바꾸면 안 되는 것

아래 자산은 권리 정리 전까지 즉시 재라이선스하면 안 된다.

### 단어 데이터와 외부 어휘 파일

점검 필요:

- 생성 스크립트로 재현 가능한가
- 외부 말뭉치와 사전에서 온 데이터가 섞여 있는가
- 상업적 재배포 제한이 있는가
- 저작권이나 사용허가가 불명확한가

현재 저장소에는 여러 단어 생성 산출물과 외부 어휘 조합 파일이 있다.
이 부분은 `출처-생성경로-재배포 가능 여부`를 먼저 표로 정리해야 한다.

### 이미지와 스크린샷

점검 필요:

- 직접 만든 이미지인가
- 외부 서비스 화면이 포함되는가
- 발표자료에 쓰인 도표와 아이콘의 출처가 명확한가

### 카카오 지도 SDK 의존 영역

점검 필요:

- 코드 자체는 라이선스 전환이 가능해도, 서비스 제공 시 Kakao 이용약관과 플랫폼 조건의 영향이 있다.
- 공익 오픈소스 방향에서는 장기적으로 지도 공급자 추상화 또는 오픈 스택 전환이 필요하다.

## 4. 권장 전환 순서

### 1단계. 자산 분류

모든 파일을 아래 네 부류로 분류한다.

- 코드
- 데이터
- 문서
- 미디어

### 2단계. 권리관계 점검

각 파일 또는 묶음마다 아래를 적는다.

- 직접 작성 여부
- 외부 출처
- 생성 스크립트 존재 여부
- 재배포 가능 여부
- 제약 사항

### 3단계. 문제 자산 격리

권리관계가 불분명한 자산은 아래 방식으로 분리한다.

- 저장소 밖으로 이동
- `private/` 또는 배포 제외 영역으로 이동
- 재현 가능한 생성 스크립트로 대체
- 완전히 새 데이터로 교체

### 4단계. 라이선스 문서 추가

필요 문서:

- 루트 `LICENSE`
- `LICENSES/` 또는 `docs/licenses/` 하위 문서
- `NOTICE`
- `TRADEMARKS.md`
- `CONTRIBUTING.md`

### 5단계. 기여 정책 정리

권장:

- CLA보다 `DCO(Sign-off)` 우선
- 모든 기여는 출처와 권리 보유를 확인한 뒤 병합
- 외부 데이터 기여는 별도 제출 양식 사용

## 5. 저장소에 적용할 권장 규칙

### 규칙 1

코드는 저장소에 들어오는 순간 AGPL 호환이 가능해야 한다.

### 규칙 2

데이터는 출처와 생성경로를 문서화하지 못하면 기본 저장소에 넣지 않는다.

### 규칙 3

문서와 템플릿은 CC BY-SA를 기본값으로 본다.

### 규칙 4

로고와 프로젝트 이름은 별도 정책 없이 방치하지 않는다.

### 규칙 5

오픈소스 전환 전에는 `현재 라이선스`, `전환 대상`, `보류 자산`을 분명히 구분해서 표시한다.

## 6. 추천하는 실제 파일 구조

예시:

```text
/LICENSE
/NOTICE
/TRADEMARKS.md
/CONTRIBUTING.md
/docs/licenses/code.md
/docs/licenses/data.md
/docs/licenses/docs.md
/data/community-packs/
/data/generated/
/data/external/
```

이 구조의 목적은 단순하다.
어떤 파일이 어떤 권리관계 아래에 있는지, 처음 온 사람도 바로 알게 하는 것이다.

## 7. 지금 시점의 현실적 결론

길마루는 바로 `완전한 카피레프트 프로젝트`라고 선언하기보다, 아래 순서가 맞다.

1. 선언문 공개
2. 권리관계 조사
3. 문제 자산 분리
4. 라이선스 전환
5. 커뮤니티 기여 체계 오픈

즉, 지금 가장 중요한 것은 `강한 오픈 라이선스 선언`보다 `재배포 가능한 자산 경계 정리`다.
